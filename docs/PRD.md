# 【完全版】プロダクト・マスター要件定義書 (PRD)

**プロジェクト名:** SaaS Idea Gravity Map & Crash Test Engine

## 1. プロダクト概要とターゲット・ペルソナ
本システムは、SaaS起業家・PMが市場の穴（需要と供給のズレ）を視覚的に発見し、LLMによる極限の検証（クラッシュテスト）を経て、強靭なビジネスモデルを構築するためのプラットフォームである。

* **ユーザーA（探索者）:** アイディアゼロ。市場の穴（重力マップ）から探索を始めたい。
* **ユーザーB（半信半疑）:** 漠然としたアイディアあり。市場マップを見て言語化し、検証へ進みたい。
* **ユーザーC（確信者）:** 明確なアイディアあり。市場調査をスキップして即座に極限検証を行いたい。
* **ユーザーD（ピボット・サバイバー）:** 過去に検証で破壊された。履歴を振り返りながら別の穴を探したい。

## 2. 全体UI/UXアーキテクチャ
「ハブ＆スポーク」モデルを採用し、すべての作業状態をホーム画面に統合・蓄積する。

* **エントリーポイント（新規アクション）:**
  * `🔭 市場の穴を探す`（Phase 1へ）
  * `🔨 アイディアを検証する`（Phase 2へ直行）
* **統合カンバンボード（ホーム画面 / The Hub）:**
  アイディアのライフサイクルを3つの列で一元管理する。
  1. `Drafts (原石)`: マップから抽出された未検証アイディア
  2. `In Validation (検証中)`: 入力途中またはAI審査中
  3. `Crushed / Survived (結果)`: 検証完了・破壊済み
* **ミクロ・ビュー（詳細パネル）:**
  カンバンのカードをクリックすると右からスライドイン。「総合生存率」「バージョン差分（V1→V2）」「AI審査員からのフィードバック履歴」を表示。

## 1. システム全体アーキテクチャ
* **コアコンセプト:** データソースに依存しない「プロバイダー抽象化レイヤー」を構築。
* **初期構成:** Product Hunt API (無料枠) を主軸とし、不足分をスクレイピングAPIで補完。
* **モデルルーティング:**
  * 開発 (`.env.local`): `gemini-3-flash-preview` (全処理)
  * 本番 (`.env.production`): Phase 1 = `gemini-3-flash-preview` / Phase 2 = `gemini-3.1-pro-preview`

## 2. データ・プロバイダー・レイヤー (Abstraction Layer)
* **目的:** 公式API、スクレイピングAPI、自前クローラーの切り替えをコード変更なしで行う。
* **取得ロジック:** 1. `Official API Mode`: Product Hunt等の公式APIを優先。
  2. `Scraping API Mode`: 公式APIがない場合、ZenRows等の有料APIで回避。
  3. `Custom Scraper Mode`: 将来的な内製化オプション。
* **レートリミット管理:** 公式APIの制限（PH: 900回/15分）を考慮し、SupabaseによるキャッシュHitを優先する。

## 3. Phase 1: 市場の穴調査 (Discover) の完全データフロー
### 3.1 引力用データ（Pull）の純化・構造化パイプライン
* **HTMLの構造的パース:** `<h1>`-`<h6>`, `<strong>` を見出しとして重み付け抽出。`<img>`, `<script>` 等は完全削除。`<ul>`, `<p>` はフラット化。
* **シグナル段落の除外:** PHのテキストから `/(launch|support us|upvote|thank you|stealth|giveaway|promo code|discount|ph community|hunters|early access)/i` を含む段落を削除。
* **タグの動的閾値と構造化:**
  * ブラックリスト（Freemium, Mac, Open Source等）を機械的に除外。
  * `Cutoff = max(2, Top_Tag_Votes * 0.05)` に満たないタグを除外。
  * 残ったタグは `[{"tag": "Kanban Board", "votes": 342}]` のJSON形式で保持し、LLMに渡す。

### 3.2 反発力用データ（Push）とJTBDデータの収集
* **動的クエリの生成タイミング（コスト削減）:** 企業ごとではなく、「ユーザーが市場を選択した瞬間」に、LLMに市場全体の『汎用的な摩擦検索クエリ』を1回だけ生成させる。
* **摩擦データの事前抽出（Lexical Filter）:** 取得したReddit等のデータから、「結合(Zapier等), 乱用・属人化(workaround等), 無消費・反逆(cancel等)」に紐づくキーワードを含む段落のみを抽出。
* **JTBD抽出（G2の回避）:** G2はスクレイピングBANリスクが高いため使用しない。代替として「AlternativeToのLikes/Dislikes」＋「PHの上位コメント」＋「摩擦コーパス」を結合してLLMに渡し、3〜5つのJTBD（星）を定義する。

### 3.3 LLMスコアリングとグローバル言語対応 (gemini-3-flash-preview)
* **ゼロコスト翻訳（Cross-lingual Hack）:** 外部翻訳API（DeepL等）は絶対に使用しない。生の英語データをGeminiに投げ、システムプロンプトで「英語で推論し、最終出力のみ日本語JSONで行うこと」を強制する。
* **引力 (Pull Score: 0-100):** 機能的解決力(40%) / 感情的達成度(40%) / 到達容易性(20%)。
* **反発力 (Push Score: 0-100):** 5つの異常行動の深刻度。

### 3.4 D3.js 物理演算・UIアニメーション仕様
* **星(JTBD):** 中央固定。大きさは需要総量（言及割合%）。
* **惑星(企業):** 距離は引力に反比例。
* **摩擦アニメーション:** ホバー時のみテンションラインを表示。反発力が高い線は赤く変化し、波打つ・振動するSVGアニメーションを付与。
* **生成トリガー:** 「💡ここでアイディアを生成」押下時のみWhat-Ifカード（4枚＋エディタドラフト）を1コールで生成。

## 4. Phase 2: アイディア極限検証 (Validate) の完全データフロー
* **入力ウィザード:** Q4付近に「業界タグ」「ITリテラシー」「マネタイズ手法」の構造化プルダウン配置。
* **逆引きマップ:** `origin === 'direct_verification'` 時、LLMがQ1/Q2からカテゴリ上位3つを提示し、選択後にPhase 1フローを非同期実行。
* **AIクラッシュエンジン (gemini-3.1-pro-preview):** The Investor, The Competitor, The Skeptical Userの3ペルソナによる並列JSON出力。
* **Git for Ideas:** 詳細パネルからピボット（再検証）時、`version` をインクリメントしV1→V2の差分を描画。

## 5. キャッシュ戦略 (Stale-While-Revalidate)
* **TTL 30日 (静的データ):** 市場共通クエリ、JTBD定義、引力用データ（公式機能）。
* **TTL 7日 (揮発性データ):** 反発力用データ（Reddit等の不満）。

## 4. Phase 2: アイディア極限検証 (Validate) の完全データフロー
### 4.1 入力・仮説構築フェーズ（状態保持と動的UI）
* **4つの問い:** Q1(Target/ビジョン), Q2(Solution), Q3(Core Value), Q4(Business/Experience)。
* **動的UI生成:** Q3のテキストからLLM(Flash)が「9つの物理要素」への言及を抽出し、Q4のタグUIを動的に最適化（ユーザーによるオーバーライド可能）。
* **修正UX:** 確認画面からQ1〜Q4のデータを保持したまま自由に行き来可能。Q1/Q2変更時のみ再要約を実行。

### 4.2 競合特定・標的選択フェーズ（Choose Your Enemy）
* **ペルソナ憑依型クエリ:** LLMが「How-to裏技」「SaaS探し」「フォーラム愚痴」の3軸で検索クエリを生成。
* **標的の選択（キメラ化防止）:** 抽出された最大3つの手段から、ユーザーが**「真っ先に倒すべき最大の標的（1つ）」**を厳格に選択する。
* **再検証時のセーフティ:** 過去の標的を使い回すか、再検索するかを選択可能。

### 4.3 評価・スコアリングフェーズ（プロスペクト理論の実装）
* **絶対評価:** LLM(Pro)が「選択された標的」と「新アイディア」の9要素を独立評価(1〜5点)。不明情報は推論せず `null` にする。
* **プロスペクト理論計算 (Next.js側):** ネガティブな差分（退化）には**3倍の損失回避バイアス**を掛ける。`null` は計算から除外し、リスク（検証の宿題）として扱う。
* **最終判定:** Total Scoreに基づき、Pivot/Proceed/Killの判定と最大の敗因を生成。

### 4.4 データベース・UI管理フェーズ（Git for Ideas）
* **一覧UI:** Pivot回数、戦っている標的、ステータスをカンバン表示。
* **ツリー型詳細UI:** 画面上部にQ1(North Star)を固定表示。直線的ではなく**「ツリー型（系統図）UI」**で派生関係(parent_version_id)を描画。
* **Semantic Diff:** バージョン間の線（エッジ）には、LLMが生成した「意味的差分（ピボット要約バッジ）」を表示。

## 4. データベース・スキーマ
* `market_cache`: `provider_type` (enum: 'ph_api', 'zenrows', 'custom') フィールドを追加し、どのソースからのデータかを識別。
